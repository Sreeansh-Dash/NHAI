# Integrating DatalakeBiometric with NHAI Data Lake 3.0

This document outlines the architecture and technical steps required to connect the `DatalakeBiometric` React Native mobile application to the existing NHAI Data Lake 3.0 platform.

## 1. Architectural Overview

NHAI's **Data Lake 3.0** is an advanced, centralized, AI-powered platform designed for end-to-end Highway Project Lifecycle Management. It aggregates data from systems like Infracon, RAMS, ATMS, and Bhoomirashi using RESTful APIs and GraphQL.

To integrate `DatalakeBiometric` as an offline-first data ingestion point for field worker attendance, the architecture relies on the app's internal **Sync Engine** connecting to the Data Lake 3.0 API Gateway.

### Integration Flow:
1. **Local Capture:** Biometric verification, timestamp, and GPS coordinates are captured completely offline and stored in the encrypted local SQLite database (`sync_outbox` table).
2. **Network Detection:** The app's `NetworkMonitor` detects when the device reaches a zone with cellular or Wi-Fi connectivity.
3. **Secure Transmission:** The `SyncManager` securely pushes batches of outbox records to the Data Lake 3.0 API via HTTPS with SSL Pinning.
4. **Ingestion & AI Analysis:** Data Lake 3.0 ingests the attendance data, making it available for payroll systems and AI-driven project progress analytics.

---

## 2. API Integration Requirements

### Endpoint Setup
NHAI must provide a dedicated RESTful POST endpoint on the Data Lake 3.0 API Gateway for attendance ingestion.
- **Example Endpoint:** `https://api.datalake.nhai.gov.in/v3/attendance/sync`

### Authentication & Authorization
The app currently uses a mock token (`mock_nhai_hackathon_token_v7`). For production integration:
1. Implement an OAuth2.0 or JWT-based login screen for the Field Manager.
2. Store the access token securely using `react-native-keychain`.
3. Inject this token into the `Authorization: Bearer <token>` header in `SyncManager.ts`.

### SSL Pinning
The app uses `react-native-ssl-pinning` to prevent Man-in-the-Middle (MitM) attacks.
1. Obtain the public certificate (`.cer` or `.der` format) for the Data Lake 3.0 API Gateway.
2. Bundle this certificate into the mobile app (in `android/app/src/main/assets/` and the iOS bundle).
3. Update `SyncManager.ts` to reference the correct certificate filename.

## 3. Data Schema Mapping

The mobile app generates attendance payloads in the following JSON structure. This must map to the Data Lake 3.0 ingestion schema.

```json
{
  "attendanceId": "uuid-v4",
  "userId": "worker-id-string",
  "timestamp": 1698765432000,
  "location": {
    "latitude": 28.5833,
    "longitude": 77.1000
  },
  "verificationScore": 0.65,
  "signature": "HMAC-SHA256:abcd1234..."
}
```

### Required Backend Validation:
- **Idempotency:** The backend must use the HTTP `X-Idempotency-Key` header (which matches `attendanceId`) to silently drop duplicate requests (returning `HTTP 409 Conflict` or `200 OK`). The mobile app's sync engine relies on this behavior to purge local records safely.
- **Audit Verification:** The backend should re-calculate the HMAC-SHA256 signature using the shared secret key to verify the payload was generated by the trusted mobile app and hasn't been tampered with.

## 4. Modifying the SyncEngine (Code Changes)

To switch from the Mock implementation to the real Data Lake 3.0 API, modify `src/sync/SyncManager.ts`:

1. Set `mockMode = false` in the `SyncManager` constructor.
2. Update `SYNC_ENDPOINT` to the official NHAI Data Lake URL.
3. Ensure the SSL pinning array matches the bundled certificate name.

```typescript
// src/sync/SyncManager.ts (Target Configuration)

const SYNC_ENDPOINT = 'https://api.datalake.nhai.gov.in/v3/attendance/sync';

// Inside syncOutbox() method:
const response = await pinnedFetch(SYNC_ENDPOINT, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`,
    'X-Idempotency-Key': record.idempotency_key
  },
  body: record.payload,
  sslPinning: {
    certs: ["nhai_api_cert"] // Must match the bundled .cer file
  }
});
```

By following these steps, the standalone biometric attendance app becomes a fully integrated, offline-capable node within the NHAI Data Lake 3.0 ecosystem.
